# fridge.ai — Project Manifest

> 가정 맞춤형 냉장고 재료 폐기 최소화 엔진
> *"버리는 재료 없이, 오늘 뭐 먹지?를 해결한다."*

- **상태:** 아이디어 / 기획 단계
- **플랫폼:** 웹 앱
- **작성일:** 2026-06-14
- **문서 버전:** 0.2 (Interview-Updated)

---

## 1. 비전 (Vision)

가정에서 발생하는 식재료 낭비를 구조적으로 줄인다. 사용자는 보유 재료를 손쉽게 등록하고,
시스템은 유통기한을 실시간으로 추적하며, 마감 임박 재료를 우선 소진할 수 있는 최적의 요리를
큐레이션한다. 매달 절약된 식비와 줄어든 탄소 발자국을 가시화함으로써
**음식물 쓰레기 감소 → 식비 절감 → 환경 기여**를 동시에 달성한다.

## 2. 문제 정의 (Problem)

- 2~3인 이상 가구는 식재료를 대량·묶음 구매하지만, 구성원 간 냉장고 현황이 공유되지 않는다.
- 냉장고 속 재료를 '망각'하여 멀쩡한 식재료를 통째로 폐기하는 일이 반복된다.
- "오늘 뭐 먹지"를 결정할 때 보유 재료 및 유통기한을 고려하지 못한다.
- 기존 레시피 앱은 콘텐츠 양은 많지만 유통기한 관리·폐기 분석 기능이 전무하다.
- 기존 메모·재고 앱은 기능이 단순해 요리 추천 연계가 불가능하다.
- 입력이 번거로워 지속 사용률이 낮다.

## 3. 핵심 가치 제안 (Value Proposition)

1. **간편 등록** — 자주 사는 재료 버튼 + 카테고리 탭 + 텍스트 검색으로 입력 마찰 최소화.
2. **D-day 자동 관리** — 유통기한 기반 D-day를 시스템이 실시간으로 추적·알림.
3. **소진 우선 큐레이션** — 마감 임박 재료를 활용하는 YouTube·블로그 레시피 추천.
4. **환경·절약 리포트** — 월간 폐기 절감량을 탄소 환산값과 절약 식비로 가시화.

## 4. 경쟁 포지셔닝 (Competitive Positioning)

| 구분 | 기존 레시피 앱 | 기존 메모 앱 | **fridge.ai** |
|------|-------------|------------|--------------|
| 레시피 | 풍부 | 없음 | YouTube·블로그 링크 |
| 유통기한 관리 | 없음 | 단순 기록 | D-day 추적 + 알림 |
| 폐기 분석 | 없음 | 없음 | 월간 절감·환경 리포트 |
| 추천 기준 | 인기·취향 | 없음 | **임박도 우선** |

### 비고객 3계층 공략 전략

- **1계층 (입력 이탈 위험군):** 자주 사는 재료 버튼으로 등록 마찰 제거
- **2계층 (앱 관리 거부층):** "오늘 안 먹으면 버리는 요리" 직관적 제시
- **3계층 (환경 무관심 미개척층):** 식비 절약·환경 기여 리포트로 신규 유입

|---|------|------|---------|

| F2 | 자주 사는 재료 버튼 | 사용자별 자주 등록하는 재료를 원터치 재등록 | P2 |

| F6 | 조리 완료·폐기 기록 | 사용 처리(재고 차감) 및 폐기 이력 기록 | P1 |

| F9 | 가정/가구 단위 공유 | 한 냉장고를 여러 구성원이 공유 | P2 |

## 6. 추천 엔진 개념 (Curation Engine)

### 레시피 소싱 전략

저작권 문제를 피하기 위해 레시피 콘텐츠를 직접 소유하지 않는다.
YouTube Data API v3를 통해 보유 재료를 검색 쿼리로 변환하여 실시간 검색 후 영상 카드로 링크한다.

```
보유 재고: [감자 D-2, 양파 D-5, 달걀 D-10]
→ 쿼리 생성: "감자 양파 달걀 요리 레시피"
→ YouTube Data API 호출
→ 영상 썸네일·제목·링크 카드 리스트 표시
```

### 스코어링 (재료 선택 우선순위)

```
score(recipe) =
      w1 * (사용 재료 중 임박 재료 비율)
    + w2 * (보유 재료 매칭률)
    + w3 * (임박도 가중합, D-day 작을수록 ↑)
    - w4 * (부족 재료 수)
```

- **MVP 전략:** 규칙 기반 쿼리 생성 → 데이터 축적 후 개인화·ML 고도화
- **장점:** DB 구축 불필요, 저작권 리스크 없음, 항상 최신 콘텐츠

## 7. 환경·절약 리포트 (Report)

감성적 톤: **환경 기여 중심** ("이번 달 탄소 1.2kg를 줄였어요")

| 항목 | 내용 |
|------|------|
| 조리 완료 기록 | 추천 요리 → 실제 조리 완료 이력 |
| 폐기 기록 | 유통기한 초과 폐기 이력 및 금액 환산 |
| 월간 절감량 | 폐기 감소 재료의 탄소 발자국 환산값 |
| 식비 절약 | 소진 재료 금액 기준 절약액 추정 |
| 환경 기여 분석 | 누적 탄소 절감 트렌드 시각화 |

## 8. 사용자 (Users)

- **1차 타깃:** 2~3인 이상 집밥형 가구 — 식재료를 대량·묶음 구매하나 구성원 간 냉장고 정보가 공유되지 않아 망각·폐기가 잦은 그룹
- **2차 타깃:** 식비·환경에 관심 있는 가정 — 절약 리포트로 유입되는 신규 사용자
- **핵심 페르소나:** "장은 봤는데 냉장고에 뭐가 있는지 가족끼리 아무도 모르고, 결국 멀쩡한 걸 버리는 사람"

## 9. 기술 방향 (Tech Direction — 잠정)

> 기획 단계이므로 확정이 아니며, 설계 단계에서 재검토한다.

- **프론트엔드:** React 기반 (Next.js 검토) — 웹 앱
- **백엔드/데이터:** API 서버 + 관계형 DB (재료·재고·폐기 이력)
- **인증:** 이메일/소셜 로그인
- **레시피 소싱:** YouTube Data API v3 (실시간 검색형)
- **추천 엔진:** 규칙 기반 쿼리 생성 모듈 (서버 사이드)
- **유통기한 데이터:** 사용자 직접 입력 (MVP) → 카테고리 프리셋 추후 추가

## 10. 범위 (Scope)

### MVP에 포함 (In Scope)
- 재료 등록 (텍스트 검색 + 카테고리 탭) (F1)
- 유통기한 D-day 추적 및 재고 대시보드 (F3–F4)
- YouTube API 기반 소진 우선 레시피 추천 (F5)

### 2차 출시 (Phase 2)
- 자주 사는 재료 버튼 (F2)
- 조리 완료·폐기 기록 (F6)
- 만료 알림 (F7)
- 식비 절약·환경 리포트 (F8)

### 초기 범위 제외 (Out of Scope — 추후)
- 가정/가구 단위 멀티유저 공유 (F9)
- 냉장고 사진 이미지 인식 자동 등록
- 바코드/영수증 스캔 자동 등록
- 유통기한 카테고리 프리셋 자동 세팅
- 마트 가격 비교·자동 장보기
- 영양 분석 / 식단 플래너
- 모바일 네이티브 앱

## 11. 비즈니스 모델 (Business Model)

- **현재:** 미정 — MVP 검증 이후 결정
- **검토 방향:** Freemium (리포트·알림·공유 기능 유료화) 또는 구독 모델

## 12. 성공 지표 (Success Metrics)

- **북극성 지표:** 사용자당 월간 폐기 재료 수 감소율
- 임박 재료 소진율 (추천 → 실제 조리 완료 전환)
- 재료 등록 지속률 (주간 활성 등록 사용자)
- 추천 레시피 클릭/조리 전환율
- 월간 환경 리포트 확인율 (3계층 유입 지표)

## 13. 로드맵 (Roadmap — 잠정)

- **Phase 0 — 기획:** 요구사항 정의, 화면 설계, 데이터 모델 (← 현재)
- **Phase 1 — MVP:** F1, F3–F5, YouTube API 추천, 단일 사용자
- **Phase 2 — 정착:** 자주 사는 재료 버튼(F2), 폐기 기록(F6), 알림(F7), 환경 리포트(F8)
- **Phase 3 — 확장:** 가구 공유(F9), 추천 개인화, 자동 등록(이미지/바코드)

## 14. 미해결 질문 (Open Questions)

- YouTube Data API v3 무료 할당량(일 10,000 유닛)이 서비스 규모에 충분한가?
- 탄소 절감량 환산 기준 데이터를 어디서 확보하나? (재료별 탄소 발자국 DB)
- 자주 사는 재료 버튼의 학습 기준은 무엇인가? (등록 빈도 기반 자동 추천 vs 사용자 수동 지정)
- 추천 정확도를 초기에 어떻게 검증할 것인가?
- 폐기 금액 환산을 위한 재료 단가 데이터를 어떻게 확보하나?
